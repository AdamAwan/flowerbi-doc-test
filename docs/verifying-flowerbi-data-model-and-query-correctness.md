# Verifying FlowerBI Data Model and Query Correctness

This guide covers practical methods to verify that your FlowerBI data model (YAML schema) and the queries you write behave as expected. Using a combination of tools built into FlowerBI, you can catch errors early, validate relationships, and confirm SQL generation matches your intentions.

## 1. Run the Interactive Playground

The [Playground](https://earwicker.com/flowerbi/demo/) lets you interactively build a query and see three things simultaneously:
- The generated JavaScript/TypeScript code for the query.
- The generated SQL that will be executed.
- The tabulated results.

This is the quickest way to verify that a specific query returns what you expect. Simply construct your `select`, `filters`, and aggregations, and inspect the SQL for correct joins, grouping, and filtering.

*Source: [FlowerBI README](https://github.com/danielearwicker/flowerbi/blob/master/README.md)*

## 2. Write Unit Tests for the Engine

FlowerBI’s core engine (`FlowerBI.Engine`) is tested using standard .NET testing frameworks (e.g., xUnit or NUnit). You can write tests that:
- Parse a YAML schema and verify that `Schema.FromYaml(...)` builds the correct model (tables, columns, foreign keys, nullability, doc/see annotations).
- Submit a query JSON and assert that the generated SQL matches an expected string (or more robustly, that executing the query against an in-memory database yields the expected rows).
- Check error handling for invalid references, missing tables, circular dependencies, etc.

Example test (pseudocode):
```csharp
[Fact]
public void CountBugsPerCustomer_GeneratesCorrectSql()
{
    var schema = Schema.FromYaml(yamlString);
    var engine = new Engine(schema);
    var queryJson = "{\n  \"select\": [\"Customer.CustomerName\"],\n  \"aggregations\": [{\"column\": \"Bug.Id\", \"function\": \"Count\"}]\n}";
    var sql = engine.BuildSql(queryJson);
    Assert.Contains("GROUP BY [Customer].[CustomerName]", sql);
}
```

*Source: [server/dotnet/FlowerBI.Engine.Tests](https://github.com/danielearwicker/flowerbi/tree/master/server/dotnet/FlowerBI.Engine.Tests)*

## 3. Inspect SQL Through Logging or Debugging

When developing, you can log the SQL generated by your API before it is executed. In your server-side code, call `Engine.BuildSql(queryJson)` and log the resulting string. This helps you verify joins, table aliases, and filter placement.

For example, the .NET `FlowerBI.Engine` exposes methods that return the SQL string. By logging it, you can spot unintended cartesian products, missing joins, or incorrect aggregate functions.

## 4. Leverage Strong Typing in TypeScript

If you use `FlowerBI.Tools` to generate TypeScript constants from your YAML schema, your client code gets auto-completion and type inference. This prevents many common mistakes:
- Misspelled column names.
- Using a column in a filter that doesn't exist.
- Mismatched data types (e.g., trying to filter a numeric column with a string).

Example generated TypeScript:
```ts
/**
 * @see {@link Topics.tenancy}
 */
export const Customer = { ... };
```
TypeScript compiler errors will catch invalid references at build time.

*Source: [docs/markdown/documentation.md](https://github.com/danielearwicker/flowerbi/blob/master/docs/markdown/documentation.md)*

## 5. Validate Queries with the `useFlowerBI` Hook State

The `useFlowerBI` React hook manages query state, including loading, errors, and data. You can inspect the state object to see if the fetch succeeded or if there was a parsing error. In a test or console, log the `state`:

```tsx
const { records, loading, error } = useFlowerBI(fetch, query);
console.log({ loading, error, records });
```

- `loading` will be `true` until the fetch resolves.
- `error` will contain any network or server-side error message.
- `records` will have the strongly-typed results.

If the schema or query is invalid, the server returns an HTTP error (e.g., 400) and the `error` field will contain the message.

*Source: [react-hooks-useflowerbi-and-usepagefilters.md](https://github.com/danielearwicker/flowerbi/blob/master/docs/markdown/react-hooks-useflowerbi-and-usepagefilters.md)*

## 6. Check Documentation Annotations in Generated Code

If you added `doc` and `see` annotations to your YAML schema, `FlowerBI.Tools` generates JSDoc/XML doc comments. These appear in your IDE’s intellisense, helping you confirm that the mapping between logical columns and their physical counterparts is correct.

Additionally, the runtime `Schema` object exposes `Table.Doc`, `Column.Doc`, and `See` lists that you can programmatically assert in tests to ensure documentation matches expectations.

*Source: [docs/markdown/documentation.md](https://github.com/danielearwicker/flowerbi/blob/master/docs/markdown/documentation.md)*

## 7. Use the Full Joins Flag to Avoid Asymmetry

When queries have multiple aggregations, the default left-join behavior can lead to unexpected missing rows. You can set `fullJoins: true` in your query to get symmetric joining. To verify, run the same query with and without the flag and compare results. This is especially important when you have filters on different aggregation columns.

*Source: [docs/markdown/full-joins.md](https://github.com/danielearwicker/flowerbi/blob/master/docs/markdown/full-joins.md)*

## 8. Validate Conjoint and Many-to-Many Relationships

If your schema uses `conjoint: true` or `associative` hints, you can write test queries that depend on these features and verify the generated SQL joins the correct tables. For conjoint tables, ensure the `@suffix` notation is consistent across `select` and `filters`. For many-to-many, confirm that the associative table is not eliminated incorrectly by the engine.

- [Conjoint Tables](https://github.com/danielearwicker/flowerbi/blob/master/docs/markdown/conjoint.md)
- [Many-to-Many](https://github.com/danielearwicker/flowerbi/blob/master/docs/markdown/many-to-many.md)

## Summary of Verification Approaches

| Method | What it validates | Speed |
|--------|-------------------|-------|
| Playground | Query SQL & results | Instant |
| Unit tests | Schema parsing, SQL generation, edge cases | Per commit |
| SQL logging | Actual generated SQL in environment | Real-time |
| TypeScript types | Client-side column/filter correctness | Build time |
| `useFlowerBI` state | Server response, error messages | Runtime |
| Doc annotation checks | Schema-to-documentation mapping | Build/Test |
| Full joins testing | Symmetry of multi-aggregation results | Per query |
| Conjoint/associative tests | Correct join logic for advanced patterns | Test suite |

By combining these techniques, you can confidently verify that your FlowerBI data model and requests behave correctly, from schema definition to final chart rendering.
